# Handoff 日常操作流程

> 公开版说明：本文已将个人路径和具体项目名改成示例占位符；请按自己的工作目录替换。

最后更新：2026-06-17  
适用场景：长窗口压缩、跨窗口接续、Codex/Claude 协同、多 Agent 分片执行  
核心原则：聊天窗口是临时工作台，`context-ledger.md` 是接力棒，本地文档是原件，知识库是搜索，`/goal` 是短目标便签。

---

## 一、为什么需要 Handoff

长任务在同一个聊天窗口里持续推进时，输入内容和模型回答都会不断累积。窗口变长后，系统可能会压缩历史上下文，导致以下问题：

- 把旧需求当成新需求；
- 忘记已经废弃的方案；
- 混淆多个项目或多个阶段；
- 新窗口无法准确接续旧窗口；
- 多 Agent 收到不同摘要后产生口径冲突。

Handoff 的目标不是阻止压缩发生，而是让关键事实不再只依赖聊天历史，而是沉淀到一个可读取、可校验、可交接的文件中。

---

## 二、核心分工

| 类型 | 定位 | 主要用途 | 风险 |
|---|---|---|---|
| 聊天窗口 | 临时工作台 | 沟通、调度、短期推理 | 会压缩、会遗忘、会混淆 |
| `context-ledger.md` | 当前任务接力棒 | 记录当前目标、有效约束、决策、下一步 | 如果不维护会过期或变垃圾堆 |
| 本地文档 | 原件 | 保存需求、方案、报告、代码说明等完整内容 | 文件多时需要入口索引 |
| 知识库 | 搜索系统 | 查找历史记录、跨会话背景、经验库 | 可能召回过期片段或噪声 |
| `/goal` | 短目标便签 | 记录当前目标、事实源路径、验收标准 | 不是无损记忆库 |
| 多 Agent | 分片执行/复核 | 并行审查、独立分析、分模块处理 | 没有统一事实源会扩大混乱 |

---

## 三、固定一个接力棒文件

每个项目建议固定一个文件：

```text
~/workspace/ai-notes/projects/项目名/context-ledger.md
```

例如某个项目可使用：

```text
~/workspace/ai-notes/projects/example-project/context-ledger.md
```

它只记录当前工作态，不要抄长文档正文。

---

## 四、`context-ledger.md` 推荐模板

```md
# Context Ledger

最后更新：2026-06-17
维护窗口：Codex / Claude / 手动

## 当前目标
- G-001：当前要完成什么。

## 当前有效约束
- R-001：必须中文为主。
- R-002：重要结论落文件，不依赖聊天历史。

## 当前有效决策
- D-001：采用 context-ledger 衔接新窗口。
- D-002：/goal 只记录短目标和事实源路径。

## 已废弃
- X-001：旧方案 A 已废弃，不再采用。

## 下一步
- N-001：下一步要做什么。

## 必读文件
- `/path/to/requirements.md`
- `/path/to/plan.md`

## 禁止事项
- 不要凭旧窗口记忆判断。
- 不要把知识库召回的旧内容直接当当前结论。
- 涉及具体文件、接口、配置时，以当前实际文件为准。
```

## 五、新窗口第一句话怎么说

新开窗口时，直接复制下面这段：

```text
请先读取 /path/to/context-ledger.md。

规则：
1. 只以其中“当前目标 / 当前有效约束 / 当前有效决策 / 已废弃 / 下一步”为任务入口；
2. 涉及具体文件、路径、接口、配置时，必须再读取实际文件确认；
3. 如果账本和实际文件冲突，以实际文件为准，并提醒我是否需要修正账本；
4. 先用 5 行复述你理解的目标、约束、下一步；
5. 复述通过后再开始执行。
```

注意：新窗口不是自动继承旧窗口，而是通过读取接力棒文件恢复上下文。

## 六、什么时候更新账本

只在以下情况更新 `context-ledger.md`：

1. 目标发生变化；
2. 关键决策发生变化；
3. 旧方案被废弃；
4. 新增重要约束；
5. 准备换窗口；
6. 准备让 Claude / Codex / 多 Agent 复核；
7. 项目进入新阶段。

普通聊天、小修小补、临时问答不要频繁更新账本，否则维护成本会过高。

---

## 七、准备换窗口时怎么收口

不要直接说“总结一下”。旧窗口可能已经受到压缩影响，直接凭记忆总结会把污染写进接力棒。

推荐指令：

```text
请先重新读取当前 context-ledger.md 和其中的必读文件。
不要凭聊天历史记忆更新。

然后起草 context-ledger.md 的更新建议，只更新：
- 当前目标
- 当前有效决策
- 已废弃
- 下一步
- 必读文件

保持总长度不超过 80 行。
先给我看 diff，不要直接大改。
```

关键规则：收口前必须重新读取账本和必读文件。

---

## 八、账本的信任边界

`context-ledger.md` 不是永远正确的真理源。

建议这样划分：

- 目标、意图、当前有效决策：优先看账本；
- 具体代码、文件路径、接口、配置：必须以当前实际文件为准；
- 账本与实际文件冲突：以实际文件为准，并修正账本；
- 知识库召回内容与账本冲突：先当背景，不直接覆盖账本；
- 账本过长或过旧：先整理再继续。

---

## 九、账本长度和维护规则

为了避免账本变成新的垃圾堆，建议遵守：

- 顶部必须写“最后更新”；
- 总长度建议不超过 80 行；
- 只保留当前有效内容和最近废弃内容；
- 长需求、长方案、长报告放到正式文档，账本只放链接或路径；
- 旧废弃内容定期移出；
- 多 Agent 默认只读账本，不并发修改账本；
- 账本更新建议由 AI 起草，人或主窗口确认后再落地。

---

## 十、`/goal` 怎么用

`/goal` 只写短目标和事实源路径。

推荐写法：

```text
/goal 当前目标：完善 handoff 日常流程。
事实源：/path/to/context-ledger.md
验收：新窗口能读取账本继续，不依赖旧聊天。
```

不要把大段需求、完整方案、长对话塞进 `/goal`。

一句话：`/goal` 防跑偏，不防遗忘；事实源仍然要落文件。

---

## 十一、多 Agent 怎么用

正确用法：

- 主窗口维护 `context-ledger.md`；
- 子 Agent 默认只读；
- 每个 Agent 都读取同一个账本；
- 派发任务时仍然要明确它负责哪一片；
- 各 Agent 输出统一模板；
- 最后由主窗口统一仲裁。

派发示例：

```text
请读取 /path/to/context-ledger.md。
你只负责审核“多 Agent 使用边界”这一节，不要修改文件。
输出：必改 / 建议 / 可选 / 风险点。
```

不要给每个 Agent 发送一段凭记忆总结的上下文，否则会把压缩错误扩散出去。

---

## 十二、什么时候开新窗口

建议开新窗口：

- 当前窗口明显很长；
- AI 开始混淆旧约束和新约束；
- 任务进入新阶段；
- 从 A 项目切到 B 项目；
- 准备做重要决策或最终交付；
- 当前窗口里混入了大量无关内容。

不需要开新窗口：

- 只是几轮小修改；
- 当前上下文还短；
- 没有出现识别混乱；
- 任务很快就结束。

## 十三、Claude 审核结论摘要

本流程已通过 Claude Code CLI 做只读审核。Claude 的总体结论是：有条件通过。

Claude 指出的关键补强点：

1. 不能盲信账本：账本管目标和决策，具体文件状态以实际文件为准；
2. 收口前必须重读：不能在已压缩污染的旧窗口里凭记忆更新账本；
3. 账本必须控长度：加最后更新时间，总长度建议不超过 80 行；
4. 多 Agent 默认只读账本，写权归主窗口；
5. 账本只引用正式文档，不复制长正文。

以上补强已合入本流程。
